# Database Schema Specification

**Database**: Neon PostgreSQL
**Version**: 1.0.0
**Created**: 2025-12-27
**Status**: Active Specification

## Overview

The database schema supports multi-user task management with conversational AI capabilities. Per constitution.md section 11, all tables include `user_id` for isolation, use UTC timestamps, and have appropriate indexes.

**Required Tables**:
1. users (managed by Better Auth)
2. tasks
3. conversations
4. messages

---

## Table 1: users

**Purpose**: Store user accounts (managed by Better Auth)

### Schema

```sql
CREATE TABLE users (
  id VARCHAR(255) PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### Columns

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | VARCHAR(255) | PRIMARY KEY | User ID (generated by auth) |
| email | VARCHAR(255) | UNIQUE, NOT NULL | User email |
| password_hash | VARCHAR(255) | - | Hashed password |
| created_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Account creation time |
| updated_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Last update time |

### Indexes

```sql
CREATE INDEX idx_users_email ON users(email);
```

### Notes

- Managed by Better Auth library
- Password never stored in plain text
- All timestamps in UTC

## Table 2: tasks

**Purpose**: Store user tasks with multi-user isolation

### Schema

```sql
CREATE TABLE tasks (
  id SERIAL PRIMARY KEY,
  user_id VARCHAR(255) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title VARCHAR(200) NOT NULL,
  description TEXT,
  due_date DATE,
  due_time TIME,
  priority VARCHAR(10) NOT NULL DEFAULT 'medium',
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  completed_at TIMESTAMP WITH TIME ZONE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT chk_priority CHECK (priority IN ('low', 'medium', 'high')),
  CONSTRAINT chk_status CHECK (status IN ('pending', 'in_progress', 'completed', 'deleted'))
);
```

### Columns

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | SERIAL | PRIMARY KEY | Auto-incrementing task ID |
| user_id | VARCHAR(255) | NOT NULL, FK → users(id) | Owner of task |
| title | VARCHAR(200) | NOT NULL | Task title |
| description | TEXT | - | Task description (optional) |
| due_date | DATE | - | Due date (optional) |
| due_time | TIME | - | Due time (optional) |
| priority | VARCHAR(10) | DEFAULT 'medium', CHECK | Task priority |
| status | VARCHAR(20) | DEFAULT 'pending', CHECK | Task status |
| completed_at | TIMESTAMP WITH TIME ZONE | - | Completion timestamp |
| created_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Creation time |
| updated_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Last update time |

### Indexes

```sql
CREATE INDEX idx_tasks_user_id ON tasks(user_id);
CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_tasks_priority ON tasks(priority);
CREATE INDEX idx_tasks_due_date ON tasks(due_date);
CREATE INDEX idx_tasks_created_at ON tasks(created_at);
CREATE INDEX idx_tasks_user_status ON tasks(user_id, status);
```

### Constraints

- **Foreign Key**: tasks.user_id → users.id (CASCADE DELETE)
- **Check**: priority IN ('low', 'medium', 'high')
- **Check**: status IN ('pending', 'in_progress', 'completed', 'deleted')

### Notes

- `user_id` required for all queries (isolation)
- `completed_at` set when status → 'completed'
- All timestamps in UTC

## Table 3: conversations

**Purpose**: Track conversation threads for stateless chat

### Schema

```sql
CREATE TABLE conversations (
  id SERIAL PRIMARY KEY,
  user_id VARCHAR(255) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### Columns

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | SERIAL | PRIMARY KEY | Auto-incrementing conversation ID |
| user_id | VARCHAR(255) | NOT NULL, FK → users(id) | Owner of conversation |
| created_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Conversation start time |
| updated_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Last message time |

### Indexes

```sql
CREATE INDEX idx_conversations_user_id ON conversations(user_id);
CREATE INDEX idx_conversations_updated_at ON conversations(updated_at);
```

### Constraints

- **Foreign Key**: conversations.user_id → users.id (CASCADE DELETE)

### Notes

- Each conversation belongs to one user
- `updated_at` set when new message added

## Table 4: messages

**Purpose**: Store conversation messages for stateless context reconstruction

### Schema

```sql
CREATE TABLE messages (
  id SERIAL PRIMARY KEY,
  user_id VARCHAR(255) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  conversation_id INTEGER NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  role VARCHAR(20) NOT NULL,
  content TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT chk_role CHECK (role IN ('user', 'assistant'))
);
```

### Columns

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | SERIAL | PRIMARY KEY | Auto-incrementing message ID |
| user_id | VARCHAR(255) | NOT NULL, FK → users(id) | Message owner |
| conversation_id | INTEGER | NOT NULL, FK → conversations(id) | Parent conversation |
| role | VARCHAR(20) | NOT NULL, CHECK | user or assistant |
| content | TEXT | NOT NULL | Message text |
| created_at | TIMESTAMP WITH TIME ZONE | DEFAULT NOW() | Message timestamp |

### Indexes

```sql
CREATE INDEX idx_messages_user_id ON messages(user_id);
CREATE INDEX idx_messages_conversation_id ON messages(conversation_id);
CREATE INDEX idx_messages_created_at ON messages(created_at);
CREATE INDEX idx_messages_conv_created ON messages(conversation_id, created_at);
```

### Constraints

- **Foreign Key**: messages.user_id → users(id) (CASCADE DELETE)
- **Foreign Key**: messages.conversation_id → conversations(id) (CASCADE DELETE)
- **Check**: role IN ('user', 'assistant')

### Notes

- Messages ordered by `created_at` for conversation history
- Composite index for efficient conversation retrieval

## Relationships

```
users (1) ──── (N) tasks
users (1) ──── (N) conversations
users (1) ──── (N) messages
conversations (1) ──── (N) messages
```

### Cascade Deletes

- Delete user → Delete all user's tasks, conversations, messages
- Delete conversation → Delete all conversation's messages

## Migrations

### Alembic Configuration

```python

# Migration naming convention
file_template = %%(year)d%%(month).2d%%(day).2d_%%(hour).2d%%(minute).2d_%%(rev)s_%%(slug)s
```

### Initial Migration

```bash
alembic revision --autogenerate -m "Initial schema: users, tasks, conversations, messages"
alembic upgrade head
```

### Rollback Support

```bash
alembic downgrade -1  # Rollback one migration
alembic downgrade base  # Rollback all migrations
```

## Data Retention

- **Tasks**: Retained until user deletes
- **Conversations**: Retained indefinitely (or 90 days inactive - TBD)
- **Messages**: Retained with conversation
- **Deleted Users**: All data cascade deleted

## Performance Considerations

### Query Patterns

Most common queries:
1. List user's tasks: `SELECT * FROM tasks WHERE user_id = ? ORDER BY created_at DESC`
2. Load conversation: `SELECT * FROM messages WHERE conversation_id = ? ORDER BY created_at ASC`
3. Update task: `UPDATE tasks SET ... WHERE id = ? AND user_id = ?`

### Index Usage

- All queries use `user_id` index
- Task queries use composite `user_id + status` index
- Message queries use composite `conversation_id + created_at` index

### Connection Pooling

- Pool size: 20 connections (default)
- Max overflow: 10 connections
- Pool recycle: 3600 seconds

## Security

### Row-Level Security (Future)

```sql
-- Example RLS policy (not implemented in v1.0)
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

CREATE POLICY tasks_user_isolation ON tasks
  FOR ALL
  USING (user_id = current_setting('app.user_id'));
```

### Prepared Statements

All queries use parameterized statements:
```sql
-- CORRECT:
SELECT * FROM tasks WHERE user_id = $1 AND id = $2;

-- WRONG (SQL injection risk):
SELECT * FROM tasks WHERE user_id = '{user_id}' AND id = {id};
```
